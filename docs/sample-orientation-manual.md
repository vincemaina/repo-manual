## What is this? (start here)

`dbt-test-lineage` answers one question about a dbt project: **"are our `not_null` / `unique` tests
doing anything, and where are we missing some?"** It does *not* run any tests or touch the warehouse. It
reads the project's column-level lineage (from the sibling **`dbt-column-lineage`** engine) plus the
tests declared in `manifest.json`, then **traces each test's guarantee through the transformations** to
decide, per column, whether the guarantee still holds. The output is a report: tests that are
**redundant** (safe to delete), columns that are **missing** coverage, and a few other lenses.

The mental model in one line: **facts in → propagate guarantees → verdicts → findings.**

## The 30-second flow

```
manifest.json (+ catalog.json)
   └─▶ [Inputs] dbt-column-lineage engine ──▶ LineageResult        (cached so re-runs are instant)
   └─▶ [Inputs] declared tests + unique_key + run_results
                         │
                         ▼
            [Propagation Core]  seed guarantees on tested columns,
                                walk the column graph, fold each transform's
                                effect → a Verdict per column (+ confidence)
                         │
                         ▼
                 [Reporting]  verdicts → findings (REDUNDANT / MISSING / UNCOVERED …)
                              + coverage, leverage, consolidation, cost
                         │
                         ▼
                    [CLI]  `report` (read it) · `check` (CI gate)
```

## ③ Propagation Core  *(the part to understand first if you change anything)*

**Purpose:** given guarantees declared on some columns and the lineage between columns, decide whether
each downstream column's guarantee still holds — soundly. This is the intellectual heart; the rest is
plumbing around it.

**What it is / why it works this way.** A test says "`orders.id` is never null." But three models
downstream that value has been through a `LEFT JOIN` and a `TRY_CAST`. Does the guarantee survive? This
system answers that by **folding the transform chain**: each transform (`rename`, `cast`, `coalesce`,
`join`…) has a known effect on a guarantee, and you compose them. The key design choice — and the reason
verdicts are trustworthy — is that it's **conservative both ways**: it only says a guarantee *holds* or
is *violated* when the facts prove it; otherwise `NOT_GUARANTEED` / `UNKNOWN`. A false "this test is
redundant, delete it" is worse than a missed one, so the whole system is biased against over-claiming.

**Members** *(source-linked — drill in):*

- [`rules.py`](../../dbt-test-lineage/src/dbt_test_lineage/rules.py) — **the effect tables.** Pure
  lookup: "what does *this one transform* do to a guarantee?"
    - `not_null_effect(step)` — maps a transform to `PRESERVE | BREAK | ESTABLISH | UNKNOWN`. The lattice
      trick: `BREAK`/`ESTABLISH`/`UNKNOWN` are *constants*, `PRESERVE` is identity → folding a chain is
      just "last non-preserve step wins." *(L60–110)*
    - `is_injective_chain(transforms)` — the uniqueness-preserving predicate (intentionally excludes
      `CAST`, since a narrowing cast can collapse distinct values). *(L130–150)*
- [`propagate.py`](../../dbt-test-lineage/src/dbt_test_lineage/propagate.py) — **the walk.** Orders the
  models topologically and pushes guarantees downstream.
    - `propagate(result, guarantees, kind)` — the entrypoint; dispatches to not_null vs unique. *(L160)*
    - `_propagate_not_null(…)` — folds each edge's chain over its upstream verdict, then **combines
      multiple edges** by the §4.3 rules (`COALESCE` = OR, `UNION` = AND). The multi-edge combination is
      the subtle bit. *(L172–230)*
    - `_propagate_unique(…)` — different shape: tracks per-model **unique key-sets** (a `GROUP BY` grain
      makes a column unique; a fan-out join breaks it). *(L300+)*
    - `column_confidence(result)` — orthogonal pass: flags verdicts that rest on *uncertain* lineage
      (an `UNKNOWN` transform, an unresolved column) so the human knows which to trust. *(L55–90)*

**How it connects:**
- **reads** the vocabulary from **① Guarantee Model** (`Verdict`, `ColumnVerdict`).
- **consumes** the `LineageResult` + declared guarantees from **② Inputs**.
- **feeds** its `{column → ColumnVerdict}` map to **④ Reporting**.

**Decisions & gotchas** *(the things that bit us — read before changing):*
- ⚠ **`NOT_GUARANTEED` ≠ `VIOLATED`.** Almost nothing is statically *provably* null (a `TRY_CAST`
  *admits* nulls without guaranteeing them). Equating the two produced 47 false alarms on a real repo.
  `VIOLATED` is reserved for genuinely provable cases. *(see `decisions/0002-not-guaranteed-vs-violated.md`)*
- ⚠ The multi-edge combination (`COALESCE`=OR, `UNION`=AND) is the easiest place to introduce a bug —
  it's why the IR carries `arg_index`/`branch` details. Covered by `tests/test_propagate.py`.
- ⚠ `unique` propagation deliberately treats fan-out joins as "*may* duplicate" (advisory), not "does."

<details><summary>▸ Drill into one function: <code>not_null_effect</code></summary>

`rules.not_null_effect(step) -> Effect` — given one `TransformStep` from the lineage, returns its effect
on a not_null guarantee. E.g. `CAST{safe:true}` (a `TRY_CAST`) → `BREAK` (yields NULL on failure);
`COALESCE` with a non-null literal default → `ESTABLISH` (not-null regardless of input); a plain
`CAST` → `PRESERVE`. A curated allowlist of null-preserving functions (`UPPER`, `TRIM`, …) maps to
`PRESERVE`; anything unrecognised → `UNKNOWN` (never a false `BREAK`).
→ [`rules.py:not_null_effect` (L60–110)](../../dbt-test-lineage/src/dbt_test_lineage/rules.py)

</details>

## ① Guarantee Model

**Purpose:** define the words. Tiny but load-bearing — every other system imports from here, so it's the
right first read.

**What it is.** Three ideas: a **`GuaranteeKind`** (`not_null` | `unique`), a **`Verdict`** lattice
(`PROVEN` / `ESTABLISHED` / `NOT_GUARANTEED` / `VIOLATED` / `UNKNOWN`, with `.holds`), and a
**`ColumnVerdict`** that pairs a column with its verdict *and the propagation path that explains it* —
so every conclusion is auditable.

**Members:** [`verdict.py`](../../dbt-test-lineage/src/dbt_test_lineage/verdict.py) — `GuaranteeKind`,
`Verdict`, `Effect`, `PropagationStep`, `ColumnVerdict`. *(the whole module, ~80 lines)*

**How it connects:** imported by everything; depends on nothing. (A "leaf" system — safe to read in
isolation.)

## ② Inputs

**Purpose:** assemble the two fact-streams the core needs — *the lineage* and *the declared tests*.

**What it is.** Two distinct sources:
- [`cache.py`](../../dbt-test-lineage/src/dbt_test_lineage/cache.py) — wraps the **upstream engine's**
  `extract_lineage` and caches the result (keyed on a hash of the manifest/catalog) so iterating on the
  report is instant. *Why it exists:* extraction is ~96% of runtime; the analysis is seconds.
- [`tests_loader.py`](../../dbt-test-lineage/src/dbt_test_lineage/tests_loader.py) — parses `manifest.json`
  test nodes into typed guarantees; plus two opt-in sources: `unique_key_guarantees` (treat a model's
  `config.unique_key` as a PK guarantee) and `load_run_results`/`load_run_metadata` (test pass/fail +
  timings, for the cost lens).

**How it connects:** `cache.py` reaches *up* to **⬡ the lineage engine**; both feed **③ Propagation Core**.

**Decisions & gotchas:**
- ⚠ `--assume-unique-key` is **opt-in** — vanilla dbt doesn't enforce `unique_key`, so treating it as a
  guarantee is only valid if your project does.
- ⚠ A `run_results.json` from `dbt docs generate` has no real test timings — the cost lens guards against
  pricing it. *(see `decisions/0003-cost-provenance-guardrail.md`)*

## ④ Reporting · ⑤ CLI  *(collapsed — expand when needed)*

<details><summary>▸ ④ Reporting — verdicts → findings</summary>

[`reports.py`](../../dbt-test-lineage/src/dbt_test_lineage/reports.py): `analyze()` turns the verdict map
into a `Report` of `Finding`s — **REDUNDANT** (tested but already proven), **REDUNDANT_STRUCTURAL** (the
model's own SQL guarantees it), **MISSING** (a guarantee got dropped, untested), **UNCOVERED** (a key with
no coverage) — plus per-kind coverage (raw + importance-weighted), test **leverage** (how far each test's
guarantee reaches), **consolidation** (which redundant tests collapse onto one anchor), and **cost** (from
run_results timings). Findings carry a `priority` and a `confidence`. *Connects:* consumes verdicts from
③; surfaced by ⑤.

</details>

<details><summary>▸ ⑤ CLI — the two commands</summary>

[`cli.py`](../../dbt-test-lineage/src/dbt_test_lineage/cli.py): `report` (human-readable / `--json`, with
`--assume-unique-key`, `--run-results`, `--cache`, `--limit`) and `check` (CI gate — exits non-zero on
provable contradictions, `--strict` also on missing). *Connects:* wires ② → ③ → ④ together; the only
system a user touches directly.

</details>

```
⬡ lineage engine ──▶ ② Inputs ──▶ ③ Propagation Core ──▶ ④ Reporting ──▶ ⑤ CLI
                          ▲                 │
```

Dependency rule worth knowing: **① is a leaf** (no deps), **⑤ is the root** (depends on everything). To
understand the repo, read in dependency order: ① → ② → ③ → ④ → ⑤. To *change* it safely, you'll want the
blast-radius lens (coming): "what else touches this function?"
